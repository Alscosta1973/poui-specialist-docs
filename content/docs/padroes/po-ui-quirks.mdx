# Quirks e Correções do PO-UI

Comportamentos não óbvios, detalhes de implementação interna e overrides de CSS
descobertos em uso de produção do PO-UI com Protheus. Aplique estas correções proativamente
ao gerar ou revisar código.

## Referência Rápida — 20 Quirks Conhecidos

| # | Componente / API | Sintoma | Correção |
|---|---|---|---|
| 1 | po-page-content | **Conteúdo invisível no carregamento com OnPush + dados síncronos** | `ngAfterViewInit() \{ setTimeout(() => this.cdr.markForCheck()); \}` |
| 2 | po-input | Botões 8px abaixo da borda do campo | `margin-bottom: 8px` no container do botão |
| 3 | po-table | Scroll horizontal em painéis lado a lado | Override da coluna checkbox para 41px via `::ng-deep`; recalcular soma dos `p-width` |
| 4 | po-input | `NG8002` em `p-max-length` | Usar `p-maxlength` (sem hífen entre `max` e `length`) |
| 5 | po-table | Barra de scroll da página aparece | Signal `tableHeight = computed(() => _winH() - OFFSET)` |
| 6 | po-table | Seleção nunca acumula | `p-selected-rows` não existe — usar eventos `(p-selected)` / `(p-unselected)` |
| 7 | po-table | Coluna checkbox congela todas as colunas | Comportamento esperado do PO-UI; usar `[p-selectable-entire-line]="false"` em telas browse |
| 8 | po-table | `$selected: false` síncrono é ignorado | Diferir a substituição do array de itens com `setTimeout(0)` |
| 9 | po-table | Sem navegação por teclado entre linhas | Padrão de 3 partes: signal `cursorIndex` + `@HostListener` + `_scrollRowIntoView` |
| 10 | po-table (duplo) | Dois browses empilhados com navegação independente por teclas | Signal `activeBrowse` + handler `Tab` redireciona setas para o browse correto |
| 11 | PoTableDetail | TS2353: `width` não existe no tipo | Remover `width` das colunas de detalhe — apenas `property`, `label`, `type`, `format` são válidos |
| 12 | po-table | **Invisível no primeiro carregamento em componente OnPush** | Sempre definir `[p-height]` — sem ele, a opacidade é definida via `setTimeout` assíncrono e o OnPush nunca re-renderiza |
| 13 | po-dynamic-form | **`(p-value-change)` silenciosamente não faz nada** | Output não existe — usar `(p-form)="fn($event)"` + assinar `form.valueChanges` |
| 14 | Signals PO-UI | Signal não desempacota automaticamente em `@Input` do PO-UI | Passar `signal()` explícito como expressão — nunca `[p-prop]="mySignal"` sem `()` |
| 15 | po-dynamic-form | `[(p-value)]` two-way binding falha silenciosamente | Não existe output `p-value-change` — usar `[p-value]` (one-way) + atualizar manualmente dentro de `onFormInit` |
| 16 | po-stepper | `[p-current-active-step]` não é `@Input` do po-stepper | API correta: `[p-step]` + `(p-change-step)`; sem `@ViewChild` |
| 17 | po-dynamic-form | `dateFormat` não existe em `PoDynamicFormField` | Usar `format` (sem prefixo `date`) |
| 18 | po-table | `(p-selected)` emite em cada ciclo de CD em browse empilhado | Spy em `(component as any).service.getItens` retornando `of([])` nos testes — evita loop HTTP infinito |
| 19 | TestBed PO-UI | `NullInjectorError: No provider for HttpClient!` mesmo sem HTTP no service | `provideHttpClient()` obrigatório em todos os TestBed — módulos PO-UI injetam HttpClient internamente |
| 20 | po-chart | `PoChartSerie.data` é `number \| number[]` | Cast explícito necessário nos specs: `(serie.data as number[]).length` |

## 1. po-page-content em branco no carregamento (timing de opacidade)

**Sintoma:** A página renderiza o título/cabeçalho mas TODO o conteúdo abaixo fica invisível no
primeiro carregamento (barra de filtros, tabelas, botões — tudo dentro de `ng-content`).

**Causa raiz (confirmada pelo código-fonte do PO-UI):**
`PoPageContentComponent` define `contentOpacity = 0` na criação e chama
`recalculateHeaderSize()` em `ngAfterViewInit`, que usa `setTimeout` para definir
`contentOpacity = 1`. Em componentes **OnPush** sem carga assíncrona de dados (ex: dados
de demo síncronos), o componente já está "limpo" quando esse `setTimeout` dispara —
o Angular pula toda a subárvore durante o CD e `contentOpacity = 1` nunca é refletido
no DOM. Componentes que fazem requisições HTTP são "acidentalmente" seguros porque a
resposta HTTP dispara um `markForCheck()` após o `setTimeout` já ter rodado.

**Afeta:** Todo componente OnPush que usa `po-page-default`, `po-page-list`,
`po-page-edit` ou `po-page-detail` E carrega dados de forma síncrona (constantes demo,
signals inicializados na declaração, etc.).

**Correção — obrigatória em todo componente OnPush que usa `po-page-*`:**

```typescript
import { AfterViewInit, ChangeDetectorRef, inject } from '@angular/core';

export class MyComponent implements OnInit, AfterViewInit {
  private readonly cdr = inject(ChangeDetectorRef);

  ngOnInit(): void {
    // carga síncrona de dados é permitida
    this.items.set(DEMO_DATA);
  }

  // po-page-content dispara seu setTimeout em seu próprio ngAfterViewInit (filho primeiro).
  // Nosso setTimeout é enfileirado depois, então quando executa contentOpacity já é 1.
  // markForCheck() força o Angular a re-verificar este componente e renderizar o conteúdo.
  ngAfterViewInit(): void {
    setTimeout(() => this.cdr.markForCheck());
  }
}
```

**Alternativa (quando HTTP é usado):** a resposta HTTP dispara `markForCheck()` automaticamente
via atualização de signal, então `ngAfterViewInit` explícito não é necessário. Mas adicioná-lo
mesmo assim é seguro e torna o componente resiliente a refatorações futuras.

```typescript
// ✗ Padrão que quebra (dados síncronos, sem correção ngAfterViewInit)
ngOnInit(): void {
  this.items.set(DEMO_DATA); // síncrono — componente nunca re-verifica após setTimeout
}

// ✓ Padrão que funciona (dados síncronos COM correção ngAfterViewInit)
ngAfterViewInit(): void {
  setTimeout(() => this.cdr.markForCheck()); // enfileirado após o setTimeout do po-page-content
}
```

## 2. po-input — desalinhamento de 8px com botões

**Sintoma:** Botões colocados na mesma linha flex que campos `po-input` aparecem 8px abaixo
da borda visível do campo, mesmo com `align-items: flex-end` no container.

**Causa raiz:** `po-input` renderiza um wrapper que inclui um espaço invisível de 8px abaixo
do campo visível, reservado para mensagens de erro de validação. O `flex-end` alinha em relação
à parte inferior desse wrapper (incluindo os 8px invisíveis), não à borda inferior do campo visível.

**Correção:** Adicionar `margin-bottom: 8px` ao elemento do botão ou ao seu container.

```scss
// ✓ Correto
.header-botoes {
  display: flex;
  align-items: center;
  gap: 6px;
  margin-left: auto;
  margin-bottom: 8px;   // ← cancela os 8px internos do po-input reservados para erros
}

.header-filtros {
  display: flex;
  align-items: flex-end;
  gap: 8px;

  po-button { margin-bottom: 8px; }  // ← mesma correção para botões dentro da linha de filtros
}
```

## 3. po-table — scroll horizontal em painéis lado a lado

**Sintoma:** Quando dois `po-table` são colocados lado a lado (flex:1 cada),
cada tabela gera uma barra de scroll horizontal mesmo com valores de `p-width` aparentemente razoáveis.

**Causa raiz:** po-table adiciona internamente uma coluna selecionável (`po-table-column-selectable`)
de **56px** que NÃO é representada em `PoTableColumn[]`. Esses 56px invisíveis são somados
aos valores de `p-width`, empurrando a tabela além dos limites do container.

**Correção (dois passos):**

**Passo 1 — Reduzir a coluna checkbox interna via `::ng-deep`:**
```scss
.browse-panel {
  ::ng-deep .po-table-column-selectable {
    width: 41px !important;
    min-width: 41px !important;
    max-width: 41px !important;
  }
}
```

**Passo 2 — Calcular os valores de `p-width` de forma que soma + checkbox = largura do painel:**
```
largura_painel = (viewport - sidebar - padding_pagina - gap_entre_paineis) / 2
orçamento_dados = largura_painel - 41px (checkbox após override)

Típico em viewport de 1366px:
  (1366 - 280 - 32 - 4) / 2 = ~520px por painel
  orçamento_dados = 520 - 41 = 479px
```

**Passo 3 — Usar gap: 4px (não 8px) entre painéis** para maximizar a largura:
```scss
.div-browses {
  display: flex;
  gap: 4px;   // não 8px — cada px economizado vai para o espaço de colunas
}
```

**Referência de largura de coluna em fonte 11px (8px de padding de célula por lado):**
| Conteúdo | Exemplo | Largura mínima |
|---|---|---|
| Data | "10/01/2025" | 84px |
| Moeda | "3.200,00" | 68px |
| Código curto 5 chars | "NF001" | 50px |
| Código médio 6 chars | "PED001" | 58px |
| Ponto de status | ● | 28px |
| Número 3 dígitos | "001" | 36px |

## 4. po-input — nome do binding `p-maxlength`

**Sintoma:** Erro de compilação Angular: `NG8002: Can't bind to 'p-max-length' since it isn't a known property of 'po-input'`.

**Causa raiz:** O alias Angular correto é `p-maxlength` (sem hífen entre `max` e `length`).

```html
<!-- ✓ Correto -->
<po-input [p-maxlength]="3" ...></po-input>

<!-- ✗ Errado — causa NG8002 -->
<po-input [p-max-length]="3" ...></po-input>
```

## 5. Altura dinâmica da tabela (sem scroll de página)

**Padrão:** Use um signal `computed` que reage ao redimensionamento da janela para manter
a altura do `po-table` preenchendo o viewport sem barra de scroll de página.

```typescript
import { AfterViewInit, HostListener, computed, signal } from '@angular/core';

// Na classe do componente:
private readonly _winH = signal(window.innerHeight);

// OFFSET = soma de todos os elementos de altura fixa fora da tabela:
//   Shell PO (toolbar + título de página + padding de conteúdo): ~232px
//   Cabeçalho específico do componente (filtros, botões): ~80px
//   Barra de título do browse: ~30px
//   Barra de legenda: ~32px
//   Rodapé de totais: ~50px
//   Total de exemplo: ~424px
readonly tableHeight = computed(() => Math.max(200, this._winH() - 424));

ngAfterViewInit(): void {
  this._winH.set(window.innerHeight);
}

@HostListener('window:resize')
onResize(): void {
  this._winH.set(window.innerHeight);
}
```

```html
<po-table [p-height]="tableHeight()" ...></po-table>
```

**Para browses de dois painéis com barra contextual condicional:**
```typescript
// Aumentar o offset pela altura da barra contextual (~36px) quando visível
readonly browseHeight = computed(() =>
  Math.max(180, this._winH() - (this.selectedRow() ? 460 : 424))
);
```

## 6. po-table — seleção: output `p-selected-rows` não existe

**Sintoma:** Sem erro TypeScript mas a seleção nunca dispara / linhas nunca acumulam.

**Causa raiz:** `p-selected-rows` não é um output de `po-table`. Os outputs corretos para
seleção de linha são eventos individuais de linha, não um array de seleção em lote.

```html
<!-- ✓ Correto -->
<po-table
  [p-selectable]="true"
  (p-selected)="onSelect($event)"
  (p-unselected)="onUnselect($event)"
  (p-all-selected)="onAllSelected($event)"
  (p-all-unselected)="selectedRows.set([])">
</po-table>

<!-- ✗ Errado — p-selected-rows não existe -->
<po-table [p-selected-rows]="selectedRows" ...></po-table>
```

## 7. po-table — colunas congeladas e `[p-selectable-entire-line]`

Quando `[p-selectable]="true"` é usado, po-table adiciona uma coluna checkbox que congela
todas as colunas (classe `po-frozen-column` em todos os `<th>`). Este é o comportamento esperado.

Para restringir a seleção apenas ao checkbox (não ao clicar em qualquer parte da linha):
```html
<po-table
  [p-selectable]="true"
  [p-selectable-entire-line]="false"
  ...>
</po-table>
```

Essencial para telas de browse de dois painéis, onde a seleção acidental de linha
ao ler dados seria prejudicial.

## 8. po-table — `$selected: false` programático não cancela seleção recém-confirmada

**Sintoma:** Quando `(p-selected)` dispara e você quer rejeitar a seleção (ex: validação de
guarda falhou), definir `$selected: false` no item de forma síncrona não tem efeito.
A linha fica visualmente selecionada, e ao clicar novamente dispara `(p-selected)` em vez de
`(p-unselected)`, fazendo a mensagem de aviso disparar uma segunda vez.

**Causa raiz:** po-table adiciona a linha ao seu array interno `selectedRows` **antes**
de despachar o evento `(p-selected)`. Qualquer alteração síncrona de `$selected: false` no
signal de itens é processada no mesmo ciclo de change detection — mas o estado interno do
po-table já tem a linha, então o estado visual permanece selecionado mesmo que o dado diga
`$selected: false`. Este dessinc causa um segundo `(p-selected)` no próximo clique em vez de `(p-unselected)`.

**Correção:** Diferir a substituição do array de itens com `setTimeout(0)`.

```typescript
// ✗ Errado — $selected: false é ignorado; po-table já confirmou a seleção
onSelectRec(item: SomeModel): void {
  if (!this.isValidSelection(item)) {
    this.notification.warning('Não é possível selecionar esta linha.');
    this.items.update(rows => rows.map(r =>
      r.id === item.id ? { ...r, $selected: false } : r
    ));
    this.cdr.markForCheck();
    return;
  }
}

// ✓ Correto — substituição diferida força po-table a reinicializar a seleção
private rejectSelection(itemId: string): void {
  setTimeout(() => {
    this.items.update(rows => rows.map(r => ({
      ...r,
      $selected: r.id === this.confirmedItem()?.id,
    })));
    this.cdr.markForCheck();
  }, 0);
}

onSelectRec(item: SomeModel): void {
  if (!this.isValidSelection(item)) {
    this.notification.warning('Não é possível selecionar esta linha.');
    this.rejectSelection(item.id);
    return;
  }
}
```

> O `setTimeout(0)` causa um flash visual de ~1 frame onde a linha aparece selecionada antes
> de reverter. Imperceptível na prática, mas inevitável com a API `(p-selected)` atual do PO-UI.

## 9. po-table — navegação por teclado entre linhas (ArrowUp / ArrowDown)

**Contexto:** po-table não tem navegação por teclado embutida. Usuários Protheus esperam
navegação com teclas de seta entre linhas (como o grid nativo do Protheus).

**Padrão — implementação em três partes:**

### Parte 1 — Estado
```typescript
readonly cursorIndex = signal<number>(0);
```

### Parte 2 — Listener de teclado + movimento do cursor
```typescript
@HostListener('window:keydown', ['$event'])
onKeyDown(event: KeyboardEvent): void {
  // Não capturar setas enquanto o usuário digita em inputs
  const tag = (event.target as HTMLElement)?.tagName?.toLowerCase();
  if (tag === 'input' || tag === 'textarea') return;

  if      (event.key === 'ArrowDown') { event.preventDefault(); this._moverCursor(1);  }
  else if (event.key === 'ArrowUp')   { event.preventDefault(); this._moverCursor(-1); }
  else return;

  // Re-aplicar highlight após o PO-UI possivelmente resetar o DOM
  setTimeout(() => this._highlightActiveRow(), 0);
}

private _moverCursor(delta: number): void {
  const items   = this.items();
  if (!items.length) return;
  const current = this.cursorIndex();
  const next    = current < 0 ? 0 : Math.max(0, Math.min(items.length - 1, current + delta));
  if (next === current && current >= 0) return;
  this.cursorIndex.set(next);
  this._onRowActivated(items[next]);
}
```

### Parte 3 — Highlight no DOM + scroll

> **⚠️ Nunca use `row.scrollIntoView(\{ block: 'nearest' \})`** — não leva em conta o
> `thead` fixo. Ao navegar para a linha 0 (ou qualquer linha próxima ao topo), a linha fica
> escondida atrás do cabeçalho fixo. Use o cálculo manual abaixo.

```typescript
private _highlightActiveRow(): void {
  document.querySelectorAll('.my-browse-container .row-ativa').forEach(
    el => el.classList.remove('row-ativa')
  );
  const idx = this.cursorIndex();
  if (idx < 0) return;
  const rows = document.querySelectorAll<HTMLElement>('.my-browse-container table tbody tr');
  const row  = rows[idx];
  if (!row) return;
  row.classList.add('row-ativa');
  this._scrollRowIntoView(row);
}

// Sobe pelo DOM a partir da linha ativa para encontrar o ancestral real com scroll.
// Não depende de nenhum nome de classe interno do PO-UI — funciona em todas as versões.
//
// Duas regras críticas:
// 1. Use setTimeout(..., 50) — NÃO 0ms. Em 0ms o DOM ainda não foi atualizado
//    pelo change detection do PO-UI, então getBoundingClientRect() retorna valores
//    obsoletos e o cálculo de scroll fica errado.
// 2. Nunca use scrollIntoView({ block: 'nearest' }): não desconta o thead fixo,
//    então a linha 0 fica escondida atrás do cabeçalho.
private _scrollRowIntoView(row: HTMLElement): void {
  let container: HTMLElement | null = row.parentElement;
  while (container && container !== document.documentElement) {
    const style = window.getComputedStyle(container);
    if (
      (style.overflowY === 'auto' || style.overflowY === 'scroll') &&
      container.scrollHeight > container.clientHeight
    ) { break; }
    container = container.parentElement;
  }
  if (!container || container === document.documentElement) return;

  const thead  = container.querySelector('thead') as HTMLElement | null;
  const theadH = thead ? thead.offsetHeight : 0;

  const cRect     = container.getBoundingClientRect();
  const rRect     = row.getBoundingClientRect();
  const rowTop    = rRect.top - cRect.top + container.scrollTop;
  const rowBottom = rowTop + row.offsetHeight;

  const visTop = container.scrollTop + theadH;
  const visBot = container.scrollTop + container.clientHeight;

  if (rowTop < visTop) {
    container.scrollTop = Math.max(0, rowTop - theadH);
  } else if (rowBottom > visBot) {
    container.scrollTop = rowBottom - container.clientHeight;
  }
}
```


### Parte 5 — Conexão com clique e inicialização
```typescript

  setTimeout(() => this._highlightActiveRow(), 0);
}

// Em ngOnInit — auto-selecionar primeira linha:
```

## 10. Navegação por teclado em browse duplo (master → detail)

Quando uma tela tem **dois po-table empilhados** onde o superior (master) direciona o inferior
(detail), ambos precisam de navegação independente. Use `activeBrowse` para rotear setas e **Tab** para alternar entre browses.

### Estado
```typescript
readonly cursorMaster = signal<number>(0);
readonly cursorDetail = signal<number>(-1);
readonly activeBrowse = signal<'master' | 'detail'>('master');
```

### Handler de teclado
```typescript
@HostListener('window:keydown', ['$event'])
onKeyDown(event: KeyboardEvent): void {
  const tag = (event.target as HTMLElement)?.tagName?.toLowerCase();
  if (tag === 'input' || tag === 'textarea') return;

  if (event.key === 'Tab') {
    event.preventDefault();
    this.activeBrowse.set(this.activeBrowse() === 'master' ? 'detail' : 'master');
    if (this.activeBrowse() === 'detail' && this.cursorDetail() < 0 && this.detailItems().length) {
      this.cursorDetail.set(0);
      setTimeout(() => this._highlightDetailRow(), 50);
    }
    return;
  }

  if (event.key !== 'ArrowDown' && event.key !== 'ArrowUp') return;
  event.preventDefault();
  const delta = event.key === 'ArrowDown' ? 1 : -1;

  if (this.activeBrowse() === 'master') {
    this._moverCursorMaster(delta);
    setTimeout(() => this._highlightMasterRow(), 50);
  } else {
    this._moverCursorDetail(delta);
    setTimeout(() => this._highlightDetailRow(), 50);
  }
}
```

### Handlers de clique — alternância do foco do browse
```typescript
onMasterClick(): void { this.activeBrowse.set('master'); }

onDetailClick(): void {
  this.activeBrowse.set('detail');
  if (this.cursorDetail() < 0 && this.detailItems().length) {
    this.cursorDetail.set(0);
    setTimeout(() => this._highlightDetailRow(), 50);
  }
}
```

### Resetar cursor do detail quando o master muda de linha
```typescript
onMasterRowSelected(row: MasterModel): void {
  const idx = this.masterItems().findIndex(i => i.id === row.id);
  this.cursorMaster.set(idx);
  this.cursorDetail.set(-1);       // ← resetar detail a cada mudança no master
  this.activeBrowse.set('master');
  this.loadDetailItems(row.id);
  setTimeout(() => this._highlightMasterRow(), 50);
}
```

### Template — wrappers com clique e indicador de browse ativo
```html
<!-- browse MASTER -->
<div class="master-browse" [class.browse-ativo]="activeBrowse() === 'master'" (click)="onMasterClick()">
  <po-table [p-columns]="masterCols" [p-items]="masterItems()" [p-selectable]="true"
    [p-selectable-entire-line]="true" [p-hide-columns-manager]="true" [p-height]="masterHeight()"
    (p-selected)="onMasterRowSelected($event)">
  </po-table>
</div>

<!-- browse DETAIL -->
<div class="detail-browse" [class.browse-ativo]="activeBrowse() === 'detail'" (click)="onDetailClick()">
  <po-table [p-columns]="detailCols" [p-items]="detailItems()" [p-selectable]="true"
    [p-hide-columns-manager]="true" [p-height]="detailHeight()"
    (p-selected)="onDetailRowSelected($event)">
  </po-table>
</div>
```

## 11. PoTableDetail — propriedade `width` não suportada (TS2353)

**Sintoma:** O compilador TypeScript lança `TS2353: Object literal may only specify known properties, and 'width' does not exist in type 'PoTableDetailColumn'`.

**Causa raiz:** `PoTableDetail.columns` usa o tipo interno `PoTableDetailColumn`, que é um
subconjunto de `PoTableColumn` e **não** inclui `width`. A propriedade `width` é válida apenas
em colunas master (`PoTableColumn[]`), não em colunas de detalhe.

**Correção:** Remover todas as propriedades `width` das colunas de detalhe. Apenas `property`, `label`, `type` e `format` são suportados.

```typescript
// ERRADO — causa TS2353
private buildDetailConfig(): PoTableDetail {
  return {
    columns: [
      { property: 'quantidade', label: 'Qtde', type: 'number', width: '8%' }, // ❌ TS2353
    ],
  };
}

// CORRETO — sem width nas colunas de detalhe
private buildDetailConfig(): PoTableDetail {
  return {
    columns: [
      { property: 'quantidade', label: 'Qtde', type: 'number', format: '1.4-4' }, // ✓
    ],
  };
}
```

## 12. po-table invisível no primeiro carregamento em OnPush (`[p-height]` obrigatório)

**Sintoma:** `po-table` renderiza corretamente no DOM mas fica invisível quando a página é
navegada pela primeira vez. Só fica visível após o usuário mover o mouse ou disparar qualquer evento.

**Causa raiz:** `po-table` inicializa `tableOpacity = 0`. Há dois caminhos para `setTableOpacity(1)`:

| Caminho | Disparado por | Timing |
|---|---|---|
| `verifyCalculateHeightTableContainer()` → `calculateHeightTableContainer()` → `setTableOpacity(1)` | `[p-height]` está definido | **Síncrono** — roda dentro de `ngDoCheck`, mesmo ciclo de CD |
| `debounceResize()` → `setTimeout(() => setTableOpacity(1))` | `offsetWidth > 0` detectado em `ngDoCheck` sem `[p-height]` | **Assíncrono** — `setTimeout` dispara após o ciclo de CD atual |

Com `ChangeDetectionStrategy.OnPush`, o componente host já está marcado como "limpo" após o
primeiro ciclo de CD. Quando o `setTimeout` assíncrono dispara e chama `setTableOpacity(1)`, o
Angular não re-verifica o componente OnPush, então a tabela fica invisível até o próximo evento.

**Correção: sempre definir `[p-height]` em todo `po-table` em componentes OnPush.**

```html
<!-- ✗ Errado — invisível no primeiro carregamento em OnPush -->
<po-table [p-columns]="cols" [p-items]="items()"></po-table>

<!-- ✓ Correto — opacidade definida sincronamente via verifyCalculateHeightTableContainer -->
<po-table [p-columns]="cols" [p-items]="items()" [p-height]="340"></po-table>
```

Para altura dinâmica (preenche o viewport), use um signal `computed` (veja Quirk 5):
```html
<po-table [p-columns]="cols" [p-items]="items()" [p-height]="tableHeight()"></po-table>
```

## 13. po-dynamic-form — `(p-value-change)` não existe

**Sintoma:** Vincular `(p-value-change)="myHandler($event)"` em `po-dynamic-form` compila
sem erros mas o handler nunca é chamado.

**Causa raiz:** `PoDynamicFormBaseComponent` declara apenas um `@Output`:
```typescript
outputs: { formOutput: "p-form" }
```
Não existe output `p-value-change`. O Angular ignora silenciosamente bindings de evento
desconhecidos em elementos customizados — sem erro de compilação, sem aviso em runtime.

**Correção: usar `(p-form)` para receber a instância do `NgForm` e assinar `form.valueChanges`.**

```typescript
import { OnDestroy } from '@angular/core';
import { Subscription } from 'rxjs';

export class MyComponent implements OnDestroy {
  headerValues: Record<string, unknown> = { adm: '46', band: 'VISA', txFixa: 1.5 };

  private formSub: Subscription | null = null;
  private _lastTx = 1.5;

  // (p-form) dispara uma vez após NgForm ser inicializado (dentro de setTimeout no fonte do PO-UI)
  onFormInit(form: any): void {
    if (!form?.valueChanges) return;
    this.formSub?.unsubscribe();
    this.formSub = form.valueChanges.subscribe((val: Record<string, unknown>) => {
      this.headerValues = { ...val };
      const tx = Number(val['myNumberField'] ?? 0);
      if (tx > 0 && tx !== this._lastTx) {
        this._lastTx = tx;
        this.doSomethingWhenFieldChanges(tx);
      }
    });
  }

  ngOnDestroy(): void {
    this.formSub?.unsubscribe();
  }
}
```

```html
<!-- ✗ Errado — (p-value-change) não existe; handler nunca é chamado -->
<po-dynamic-form [p-fields]="fields" [p-value]="vals" (p-value-change)="onChange($event)">
</po-dynamic-form>

<!-- ✓ Correto — (p-form) é o único output reativo; assinar form.valueChanges -->
<po-dynamic-form [p-fields]="fields" [p-value]="headerValues" (p-form)="onFormInit($event)">
</po-dynamic-form>
```

> `form.valueChanges` dispara a **cada tecla** em campos `po-decimal` / `po-number` e no **blur** para campos de texto.

## 18. po-table — `(p-selected)` emite em cada ciclo de CD (browse empilhado)

**Sintoma (em testes):** `httpMock.verify()` falha com uma requisição GET `/detail` pendente após
cada flush de HTTP no spec de browse empilhado, criando um loop infinito de requisições.

**Causa raiz:** Com `[p-selectable]="true"`, o `po-table` emite `(p-selected)` em cada ciclo
de change detection — não apenas nas interações do usuário. Em browses empilhados onde a
seleção do master dispara um GET de detail, cada flush HTTP força um `markForCheck()` → novo
ciclo de CD → nova emissão de `(p-selected)` → novo GET.

**Correção nos testes:** Fazer spy no método `getDetail` do service para retornar `of([])`,
impedindo que qualquer requisição HTTP seja criada mesmo durante os ciclos automáticos do po-table:

```typescript
beforeEach(() => {
  // Impede o loop: po-table emite (p-selected) em cada CD,
  // mas agora não gera requisição HTTP
  spyOn((component as any).service, 'getItens').and.returnValue(of([]));
});
```

## 19. TestBed — `provideHttpClient()` obrigatório mesmo sem HTTP no service

**Sintoma:** `NullInjectorError: No provider for HttpClient!` em testes de componentes cujos
services não fazem nenhuma chamada HTTP.

**Causa raiz:** Os módulos PO-UI (`PoPageModule`, `PoTableModule`, etc.) injetam `HttpClient`
internamente para carregar ícones e recursos — independentemente de o service do componente
usar HTTP ou não.

**Correção:** Sempre incluir ambos em todos os TestBed de componentes PO-UI:

```typescript
TestBed.configureTestingModule({
  imports: [MeuComponente],
  providers: [
    provideHttpClient(),          // ← obrigatório sempre
    provideHttpClientTesting(),   // ← permite httpMock.verify() sem erros
    provideRouter([]),
  ],
});
```
