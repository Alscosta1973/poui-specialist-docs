1. po-page-content em branco no carregamento (timing de opacidade)

Sintoma: A página renderiza o título/cabeçalho mas TODO o conteúdo abaixo fica invisível no primeiro carregamento (barra de filtros, tabelas, botões — tudo dentro de ng-content).

Causa raiz (confirmada pelo código-fonte do PO-UI): PoPageContentComponent define contentOpacity = 0 na criação e chama recalculateHeaderSize() em ngAfterViewInit, que usa setTimeout para definir contentOpacity = 1. Em componentes OnPush sem carga assíncrona de dados (ex: dados de demo síncronos), o componente já está "limpo" quando esse setTimeout dispara — o Angular pula toda a subárvore durante o CD e contentOpacity = 1 nunca é refletido no DOM. Componentes que fazem requisições HTTP são "acidentalmente" seguros porque a resposta HTTP dispara um markForCheck() após o setTimeout já ter rodado.

Afeta: Todo componente OnPush que usa po-page-default, po-page-list, po-page-edit ou po-page-detail E carrega dados de forma síncrona (constantes demo, signals inicializados na declaração, etc.).

Correção — obrigatória em todo componente OnPush que usa po-page-*:

Alternativa (quando HTTP é usado): a resposta HTTP dispara markForCheck() automaticamente via atualização de signal, então ngAfterViewInit explícito não é necessário. Mas adicioná-lo mesmo assim é seguro e torna o componente resiliente a refatorações futuras.

2. po-input — desalinhamento de 8px com botões

Sintoma: Botões colocados na mesma linha flex que campos po-input aparecem 8px abaixo da borda visível do campo, mesmo com align-items: flex-end no container.

Causa raiz: po-input renderiza um wrapper que inclui um espaço invisível de 8px abaixo do campo visível, reservado para mensagens de erro de validação. O flex-end alinha em relação à parte inferior desse wrapper (incluindo os 8px invisíveis), não à borda inferior do campo visível.

Correção: Adicionar margin-bottom: 8px ao elemento do botão ou ao seu container.

3. po-table — scroll horizontal em painéis lado a lado

Sintoma: Quando dois po-table são colocados lado a lado (flex:1 cada), cada tabela gera uma barra de scroll horizontal mesmo com valores de p-width aparentemente razoáveis.

Causa raiz: po-table adiciona internamente uma coluna selecionável (po-table-column-selectable) de 56px que NÃO é representada em PoTableColumn[]. Esses 56px invisíveis são somados aos valores de p-width, empurrando a tabela além dos limites do container.

4. po-input — nome do binding p-maxlength

Sintoma: Erro de compilação Angular: NG8002: Can't bind to 'p-max-length' since it isn't a known property of 'po-input'.

Causa raiz: O alias Angular correto é p-maxlength (sem hífen entre max e length).

6. po-table — seleção: output p-selected-rows não existe

Sintoma: Sem erro TypeScript mas a seleção nunca dispara / linhas nunca acumulam.

Causa raiz: p-selected-rows não é um output de po-table. Os outputs corretos para seleção de linha são eventos individuais de linha, não um array de seleção em lote.

7. po-table — colunas congeladas e [p-selectable-entire-line]

Quando [p-selectable]="true" é usado, po-table adiciona uma coluna checkbox que congela todas as colunas (classe po-frozen-column em todos os <th>). Este é o comportamento esperado.

8. po-table — $selected: false programático não cancela seleção recém-confirmada

Sintoma: Quando (p-selected) dispara e você quer rejeitar a seleção (ex: validação de guarda falhou), definir $selected: false no item de forma síncrona não tem efeito. A linha fica visualmente selecionada, e ao clicar novamente dispara (p-selected) em vez de (p-unselected), fazendo a mensagem de aviso disparar uma segunda vez.

Causa raiz: po-table adiciona a linha ao seu array interno selectedRows antes de despachar o evento (p-selected). Qualquer alteração síncrona de $selected: false no signal de itens é processada no mesmo ciclo de change detection — mas o estado interno do po-table já tem a linha, então o estado visual permanece selecionado mesmo que o dado diga $selected: false. Este dessinc causa um segundo (p-selected) no próximo clique em vez de (p-unselected).

Correção: Diferir a substituição do array de itens com setTimeout(0).

O setTimeout(0) causa um flash visual de ~1 frame onde a linha aparece selecionada antes de reverter. Imperceptível na prática, mas inevitável com a API (p-selected) atual do PO-UI.

11. PoTableDetail — propriedade width não suportada (TS2353)

Sintoma: O compilador TypeScript lança TS2353: Object literal may only specify known properties, and 'width' does not exist in type 'PoTableDetailColumn'.

Causa raiz: PoTableDetail.columns usa o tipo interno PoTableDetailColumn, que é um subconjunto de PoTableColumn e não inclui width. A propriedade width é válida apenas em colunas master (PoTableColumn[]), não em colunas de detalhe.

Correção: Remover todas as propriedades width das colunas de detalhe. Apenas property, label, type e format são suportados.

12. po-table invisível no primeiro carregamento em OnPush ([p-height] obrigatório)

Sintoma: po-table renderiza corretamente no DOM mas fica invisível quando a página é navegada pela primeira vez. Só fica visível após o usuário mover o mouse ou disparar qualquer evento.

Causa raiz: po-table inicializa tableOpacity = 0. Há dois caminhos para setTableOpacity(1):

CaminhoDisparado porTiming
verifyCalculateHeightTableContainer() → calculateHeightTableContainer() → setTableOpacity(1)[p-height] está definidoSíncrono — roda dentro de ngDoCheck, mesmo ciclo de CD
debounceResize() → setTimeout(() => setTableOpacity(1))offsetWidth > 0 detectado em ngDoCheck sem [p-height]Assíncrono — setTimeout dispara após o ciclo de CD atual

Com ChangeDetectionStrategy.OnPush, o componente host já está marcado como "limpo" após o primeiro ciclo de CD. Quando o setTimeout assíncrono dispara e chama setTableOpacity(1), o Angular não re-verifica o componente OnPush, então a tabela fica invisível até o próximo evento.

Correção: sempre definir [p-height] em todo po-table em componentes OnPush.

13. po-dynamic-form — (p-value-change) não existe

Sintoma: Vincular (p-value-change)="myHandler($event)" em po-dynamic-form compila sem erros mas o handler nunca é chamado.

Causa raiz: PoDynamicFormBaseComponent declara apenas um @Output:

Não existe output p-value-change. O Angular ignora silenciosamente bindings de evento desconhecidos em elementos customizados — sem erro de compilação, sem aviso em runtime.

Correção: usar (p-form) para receber a instância do NgForm e assinar form.valueChanges.

form.valueChanges dispara a cada tecla em campos po-decimal / po-number e no blur para campos de texto.

14. Signals PO-UI — não desempacotam automaticamente em @Input

Sintoma: [p-title]="mySignal" exibe [object Object] ou Signal(valor) em vez do valor real.

Causa raiz: Inputs do PO-UI não são input() do Angular — são @Input() decorators clássicos. O Angular não desempacota signals automaticamente nesses casos.

Correção: Sempre invocar o signal com ():

15. po-dynamic-form — [(p-value)] two-way binding falha silenciosamente

Sintoma: Alterações no formulário não atualizam a variável vinculada; o valor sempre permanece o inicial.

Causa raiz: Não existe output p-value-change no PO-UI (veja Quirk 13). Sem o output, o Angular não consegue completar o two-way binding.

Correção: Usar [p-value] (one-way) para inicialização e atualizar o objeto manualmente dentro de onFormInit:

16. po-stepper — API de step incorreta

Sintoma: [p-current-active-step] não funciona; erro de binding ou step não muda.

Causa raiz: [p-current-active-step] não é um @Input do po-stepper. A API correta usa [p-step] + (p-change-step).

Correção: Usar signal currentStep + bindings corretos, sem @ViewChild:

17. po-dynamic-form — dateFormat não existe em PoDynamicFormField

Sintoma: Erro de compilação TypeScript: Object literal may only specify known properties, and 'dateFormat' does not exist.

Causa raiz: A propriedade correta é format, sem o prefixo date.

18. po-table — (p-selected) emite em cada ciclo de CD (browse empilhado)

Sintoma (em testes): httpMock.verify() falha com uma requisição GET /detail pendente após cada flush de HTTP no spec de browse empilhado, criando um loop infinito de requisições.

Causa raiz: Com [p-selectable]="true", o po-table emite (p-selected) em cada ciclo de change detection — não apenas nas interações do usuário. Em browses empilhados onde a seleção do master dispara um GET de detail, cada flush HTTP força um markForCheck() → novo ciclo de CD → nova emissão de (p-selected) → novo GET.

Correção nos testes: Fazer spy no método getDetail do service para retornar of([]), impedindo que qualquer requisição HTTP seja criada mesmo durante os ciclos automáticos do po-table:

19. TestBed — provideHttpClient() obrigatório mesmo sem HTTP no service

Sintoma: NullInjectorError: No provider for HttpClient! em testes de componentes cujos services não fazem nenhuma chamada HTTP.

Causa raiz: Os módulos PO-UI (PoPageModule, PoTableModule, etc.) injetam HttpClient internamente para carregar ícones e recursos — independentemente de o service do componente usar HTTP ou não.

Correção: Sempre incluir ambos em todos os TestBed de componentes PO-UI:

20. po-chart — PoChartSerie.data é number | number[]

Sintoma (em testes): Erro TypeScript TS2339: Property 'length' does not exist on type 'number | number[]' ao acessar .length em serie.data.

Causa raiz: O tipo PoChartSerie declara data como number | number[], então o TypeScript não permite acessar .length diretamente.

Correção: Usar cast explícito nos specs:
